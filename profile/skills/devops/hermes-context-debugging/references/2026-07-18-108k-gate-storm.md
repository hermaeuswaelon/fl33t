# 108K Gate Storm — Session 2026-07-18

## Reproduction

Profile: `thotheauphis`  
Model: `deepseek-reasoner` via `deepseek` provider  
Context: 128K parent / 32K delegate (hard caps)  
Compression thresholds: 50K/88K/108K (three-tier)

### Observed Behavior

```
errors.log: 489× "108K Gate: ~111,995 tokens >= 108,000 — forcing compression"
```

Compression fired every turn. Token count stayed at ~112K because the **incompressible prompt floor** (system prompt + skills index ~10K + tool schemas ~26K + memory + user context) alone exceeded the 108K threshold. Each compression pass dropped assistant/tool messages, but the floor kept the next turn's request above 108K.

### Debugging Path

1. Searched `errors.log` for "108K Gate" → 489 hits
2. Traced to `tools/offload_system.py` `gate_should_stop()` — fires when `request_pressure_tokens >= 108000`
3. Found gate override in `conversation_loop.py` line 1024 — bypasses `should_compress()` entirely
4. `compression_attempts < 3` was per-turn only (resets each turn)
5. `context_compressor.py` `should_compress()` had no once-only tracking

### Two-Part Fix

**Part 1 — context_compressor.py:**
- Added `_has_fired_tier1`, `_has_fired_tier1_5`, `_has_fired_tier2` flags in `__init__`
- Each flag set to `True` when that tier fires in `should_compress()`
- Returns `False` if flag is already `True` for the matching tier
- Flags are session-level (never reset)

**Part 2 — conversation_loop.py:**
- Gate override condition: added `not getattr(_compressor, '_gate_has_fired', False)`
- After `_compress_context()` returns: set `_compressor._gate_has_fired = True`
