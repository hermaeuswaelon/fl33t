## Architecture Notes

### Parameter Profile Hierarchy
```
ARES offloader/continuity daemons → ares-param (Pascal/CLI, 38K token decay)
                                    ↓ different frequencies
Hermes goal runner → parameter_control_tool (Python, manual switching)
```

Both systems coexist. ARES handles daemon parameter injection; this session's tool handles interactive goal-driven tuning.

### Dual-Path Pattern (Skill + Tool)
Both `ctx_curation` and `goal_runner` follow the same pattern:
- **Skill path** (`/ctx-curation`, `/goal` future): loads full guidance document (~2KB), good for first use
- **Tool path** (`ctx_curate()`, `goal_runner()`/`goal_turn()`): zero document overhead, direct function calls, same persisted state

### MCP Server Design
File-backed = zero infrastructure. SQLite + JSONL + JSON survives reboots, no daemon dependencies. Tools expose:
- **forge**: write/read/search/list_namespaces (agent memory)
- **fleet**: status/list_binaries/run_binary (Pascal fleet control)
- **context**: compress/domains (context curation)
- **memory**: store/recall/list_namespaces (encrypted agent_memory)
- **browser**: ping/navigate/execute (CEF via socket)
- **event**: publish/list_topics (Thoth bus)
- **sensor**: netlens/memlens/procsight (sensor daemons)
- **info**: server metadata

All wired via `config.yaml` mcp_servers section.
